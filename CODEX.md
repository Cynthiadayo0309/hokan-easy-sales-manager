# Codex 実装指示書

## 1. プロジェクト情報

- アプリ名: **訪看かんたん売上管理**
- 英語プロジェクト名: **Hokan Easy Sales Manager**
- 推奨リポジトリ名: `hokan-easy-sales-manager`
- 対象OS: Windows 10 / 11（64bit）
- 提供形式: Windows用インストーラー（`.exe`）
- 利用方式: ローカルデスクトップアプリ
- インターネット接続: 不要
- 初期利用者: 管理者1名

### アプリの一言説明

5施設の訪問看護実績を週に1回入力し、医療・精神科・介護の売上見込みと月間目標達成率を簡単に確認するWindowsアプリ。

## 2. Codexへの最優先指示

このリポジトリでは、以下を最優先とする。

1. **機械操作に不慣れな人でも迷わず使えること**
2. **インストーラーを実行すれば、追加設定なしですぐ使えること**
3. **オフラインで動作し、データをローカル端末へ保存すること**
4. **週次入力から月内実績だけを正しく集計すること**
5. **既存データを壊さずにアップデートできること**
6. **将来の複数利用者・共有データベース化に備え、画面・計算・保存処理を分離すること**

見た目の派手さよりも、分かりやすさ、入力ミスの防止、データ保全を優先すること。

## 4. 採用技術

以下を基本構成とする。

- Electron
- Vue 3
- TypeScript
- Vite
- Pinia
- Vue Router
- SQLite
- better-sqlite3
- electron-builder
- Vitest
- Playwright（Electronの主要操作確認に使用できる場合）

大きなUIライブラリは必須ではない。導入する場合も、画面を複雑にしないこと。

## 5. MVPの対象範囲

初期バージョンで実装する。

- 5施設の登録・編集
- 医療訪問看護、精神科訪問看護、介護訪問看護の管理
- 1回・2回・3回訪問別の売上単価設定
- 対象年月ごとの目標人数、目標件数、目標売上設定
- 日曜日始まり、土曜日終わりの期間生成
- 月初・月末は月内の日付だけに分割
- 週1回、5施設分の実績入力
- 人数から訪問件数と売上見込みを自動計算
- 施設別、看護区分別、期間別、5施設合計の月間集計
- 月間目標達成率、目標差額、月末予測
- 入力済み・未入力の管理
- 前期間コピー
- 自動保存または安全な一時保存
- 過去月の閲覧
- 月の締め処理
- CSV出力
- バックアップ・復元
- Windowsインストーラー作成
- 上書きアップデート時のデータ保持

## 6. MVPで実装しないもの

以下は将来拡張とし、初期バージョンには含めない。

- クラウド同期
- 複数端末でのリアルタイム共有
- 利用者ごとのログイン画面
- メール通知
- 外部サービス連携
- 自動アップデート配信サーバー
- 医療保険・介護保険請求システムとの連携
- 個人を特定する患者情報の登録

ただし、将来追加しやすい内部設計にすること。

## 7. 実装上の禁止事項

- RendererからSQLiteを直接操作しない
- `nodeIntegration: true` にしない
- `contextIsolation: false` にしない
- データベースをアプリのインストールフォルダーへ保存しない
- 金額をJavaScriptの浮動小数だけで曖昧に管理しない
- 月をまたぐ1週間を、どちらか片方の月へ丸ごと計上しない
- UIに技術用語、英語エラー、スタックトレースを表示しない
- 施設名や単価をソースコードへ固定しない
- アプリ更新時に既存DBを初期化しない
- ネットワーク接続を必須にしない
- 患者氏名、住所、病名などの個人情報を保存しない

## 8. 金額の扱い

画面上の金額単位は **千円** とする。

例:

- `500千円` = 500,000円
- `1,250千円` = 1,250,000円

内部では誤差を避けるため、金額を **円の整数値** で保持することを推奨する。画面入出力時に千円へ変換する。

- DB保存: 1,250,000円
- 画面表示: 1,250千円

小数の千円入力を許可する場合、`8.5千円` を `8,500円` として整数変換すること。

## 9. Codexの作業方法

1. 最初に既存リポジトリの状態を確認する。
2. 変更前に、実装予定を短いチェックリストで提示する。
3. 1機能ずつ、小さな単位で実装する。
4. 各機能にテストを追加する。
5. DB変更には必ずマイグレーションを追加する。
6. 既存仕様を変更する場合は、理由と影響範囲を説明する。
7. 実装後は、実行したテストと結果を報告する。
8. Windowsインストーラーが生成できるところまで確認する。

不明点があっても実装を止めず、本ドキュメントの既定値で進める。業務結果が変わる重大な不明点だけを `docs/OPEN_QUESTIONS.md` に記録すること。

## 10. Definition of Done

初期バージョンは、以下をすべて満たしたとき完了とする。

- Windows用インストーラーを生成できる
- 利用者はNode.jsやSQLiteを別途インストールしなくてよい
- 初回起動時にDBが自動作成される
- 5施設を順番に入力できる
- 入力対象年月・期間・施設が常に分かる
- 日曜始まりの期間が正しく生成される
- 月初・月末の部分期間が正しく生成される
- 月をまたぐ実績がそれぞれの月へ分かれる
- 人数、件数、売上、達成率が正しく計算される
- 金額表示に「千円」が明記される
- 未入力施設・期間が分かる
- 前期間コピー後に確認して保存できる
- 過去月を閲覧できる
- バックアップと復元が画面操作でできる
- アプリを更新しても入力データが残る
- オフラインで主要機能がすべて動作する
- 主要計算ロジックの自動テストが成功する
- 機械操作に不慣れな利用者向けの日本語エラー表示になっている
